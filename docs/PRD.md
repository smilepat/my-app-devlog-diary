# PRD (Product Requirements Document)

## 1) 문제 정의 (Why)
- 필드가 너무 많아질 우려가 있음
  - 데이터 테이블 설계 시 필드 수가 늘어나면 관리가 복잡해지고, 사용자 경험도 저하될 수 있음

## 2) 목표 사용자 (Who)
> 현재 로그 기준으로는 정보 부족
>
> 💡 다음 로그에 "이 앱을 누가 사용하는지 (예: 개발자, 일반 사용자, 학습자 등)"를 기록하면 좋겠습니다.

## 3) 목표/성공지표 (Goals)
> 현재 로그 기준으로는 정보 부족
>
> 💡 다음 로그에 "이 프로젝트의 성공 기준 (예: 사용자 수, 기능 완성도 등)"을 기록하면 좋겠습니다.

## 4) 핵심 기능 (What)
- 어휘 데이터 테이블 구조 설계 완료
- Google Sheets 연동 방식 결정
  - 스프레드시트를 데이터베이스처럼 활용하여 접근성을 높임

## 5) 데이터/연동 구조 (How)
- **Google Sheets** 기반 데이터 저장소 사용
- **Apps Script**로 CRUD API 구현 예정
  - CRUD: 데이터의 생성(Create), 읽기(Read), 수정(Update), 삭제(Delete) 기능
- MVP에서는 **8개 필드**만 유지하여 단순화

## 6) 범위 (MVP / 다음 단계)
### MVP (최소 기능 제품)
- 8개 필드로 구성된 어휘 데이터 테이블
- Google Sheets 연동

### 다음 단계
- Apps Script로 CRUD API 설계 및 구현

## 7) 현재 상태 & 다음 할 일
### 현재 상태 (2025-12-23 기준)
- ✅ 어휘 데이터 테이블 구조 설계 완료
- ✅ Google Sheets 연동 방식 결정 완료
- ✅ MVP 범위 확정 (8개 필드)

### 다음 할 일
- [ ] Apps Script로 CRUD API 설계
- [ ] API 엔드포인트 정의
- [ ] 테스트 데이터로 연동 검증
